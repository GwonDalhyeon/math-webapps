# 수학 웹앱 제작 지침

이 문서는 새 수학 웹앱을 만들거나 기존 웹앱의 핵심 동작을 수정할 때 읽는다.

## 기본 원칙

이 프로젝트의 웹앱은 계산기보다 "수학 개념 실험실"에 가깝다.

좋은 웹앱은 다음을 보여준다.

- 값이 바뀔 때 무엇이 함께 변하는가
- 결과가 만들어지는 과정
- 누적, 수렴, 분포 형성, 비교의 변화
- 그래프나 도식과 수치 해석의 연결
- 교사가 짧게 시연해도 학생이 핵심을 읽을 수 있는 흐름

피해야 할 것:

- 결과 숫자만 출력하는 앱
- 인터랙션이 거의 없는 앱
- 의미가 불분명한 그래프
- 한 화면에 너무 많은 그래프를 동시에 넣는 구성
- 보기만 화려하고 수학적 의미가 약한 효과

## 기술 기준

기본 기술:

- HTML
- CSS
- 브라우저 JavaScript
- Canvas
- SVG
- MathML
- 정적 이미지, 오디오, JSON, CSV

조건부 사용:

- 복잡한 수식 렌더링이 필요할 때만 MathJax CDN을 사용한다.
- 외부 CDN을 쓰면 인터넷 연결이 필요하다는 점을 고려한다.
- JavaScript 모듈을 사용할 때는 상대경로로 동작해야 한다.

금지:

- React, Vite, Next.js, TypeScript, 번들러
- Node.js 런타임을 전제로 한 구조
- npm 설치나 빌드 과정
- 서버 코드, 데이터베이스, 로그인, 서버 API
- 환경변수
- 로컬 절대경로

## 화면 흐름

가능하면 다음 흐름을 따른다.

```text
설명 -> 조작 -> 변화 확인 -> 해석
```

필수에 가까운 화면 요소:

1. 짧은 개념 설명
2. 핵심 시각화 영역
3. 사용자 조작 영역
4. 실행 버튼
5. 현재 상태 또는 결과 해석 영역

반복 실험, 누적, 분포 형성 앱의 권장 버튼:

- 1회 실행
- 단계 실행
- 10회 실행
- 100회 실행
- 자동 실행
- 일시정지
- 초기화

개념상 불필요한 버튼은 생략해도 된다.

## 학습 자료 다운로드

웹앱에 맞는 학습지를 함께 제작했거나 해당 웹앱 폴더에 학습지 파일이 있는 경우, 특별한 지시가 없어도 웹앱 안에 다운로드 영역을 만든다.

- 위치는 제목/짧은 설명 아래, 조작 영역 위가 기본이다.
- 핵심 시각화와 조작 영역을 방해하지 않도록 얇은 자료 바 또는 접이식 메뉴로 둔다.
- 자료가 1~2개이면 바로 보이는 작은 버튼으로 둔다.
- 자료가 3개 이상이면 `<details>`와 `<summary>`로 접어 "자료 다운로드" 한 줄만 기본 노출한다.
- PDF와 Markdown 원본이 함께 있으면 둘 다 연결한다.
- 학생용과 교사용이 함께 있으면 둘 다 연결한다.
- 실제 파일이 없는 링크는 만들지 않는다.

예:

```html
<details class="resource-menu">
  <summary>자료 다운로드</summary>
  <div class="resource-list">
    <a href="./worksheet.pdf" download>학생용 PDF</a>
    <a href="./worksheet-teacher.pdf" download>교사용 PDF</a>
    <a href="./worksheet.md" download>학생용 MD</a>
    <a href="./worksheet-teacher.md" download>교사용 MD</a>
  </div>
</details>
```

## 수학 시각화 기준

항상 다음 질문에 답할 수 있어야 한다.

- 사용자가 값을 바꾸면 무엇이 즉시 변하는가?
- 그래프, 도식, 표, 수치, 설명 문구가 함께 반응하는가?
- 학생이 현재 상태를 한눈에 읽을 수 있는가?
- 최종 결과만이 아니라 중간 과정이나 누적 변화가 보이는가?
- 그래프는 왜 필요한가?
- 교사가 수업 중 짧게 시연하기에 충분히 단순한가?

그래프를 사용할 때:

- 축, 단위, 기준선, 색의 의미를 분명히 표시한다.
- 색만으로 의미를 구분하지 말고 라벨이나 문구를 함께 둔다.
- 핵심 변화가 잘 보이도록 그래프 수를 줄인다.
- 최종 결과보다 변화 과정, 비교, 누적, 수렴, 분포 형성을 보여준다.

## 수학 표현

- 복잡한 수식은 LaTeX 문법으로 작성하고, MathJax 또는 MathML로 렌더링한다.
- 단순한 인라인 수식은 텍스트 표기로 충분한지 먼저 판단한다.
- 분수, 극한, 합, 적분, 확률식처럼 구조가 중요한 수식은 LaTeX 렌더링을 우선 고려한다.
- 변수, 매개변수, 함수 이름, 확률기호, 극한기호는 한 웹앱 안에서 일관되게 사용한다.
- 수식만 나열하지 말고 그래프, 도식, 표, 현재 상태 문구와 연결한다.

## 디자인 기준

- 공통 스타일은 가능한 한 `../../assets/common.css`를 재사용한다.
- 개별 웹앱에만 필요한 스타일은 해당 `index.html`의 `<style>` 안에 둘 수 있다.
- 교실 프로젝터에서 잘 보이는 큰 글씨와 높은 대비를 우선한다.
- 조작 영역과 시각화 영역을 명확히 분리한다.
- 버튼, 슬라이더, 숫자 표시는 충분히 크게 만든다.
- 모바일에서도 기본 조작이 깨지지 않도록 반응형으로 배치한다.
- 페이지 섹션을 과도하게 카드처럼 중첩하지 않는다.
- 장식보다 수학적 변화의 가독성을 우선한다.

## 코드 기준

- 전체 코드는 바로 실행 가능한 완성본으로 작성한다.
- HTML, CSS, JavaScript를 읽기 쉽게 작성한다.
- 변수명과 함수명은 의미가 드러나게 한다.
- 수학 계산 로직, 상태 관리, 화면 갱신 로직을 가능하면 분리한다.
- 입력값 오류와 범위 초과를 고려한다.
- 자동 실행, 일시정지, 초기화에서 상태가 꼬이지 않게 한다.
- 자동 실행 중 중복 타이머가 생기지 않게 한다.
- 주석은 복잡한 계산이나 상태 전환을 설명할 때만 짧게 단다.

## 새 웹앱 제작 순서

1. 학습 목표를 짧게 정리한다.
2. 화면 구성을 간단히 설계한다.
3. 필요한 파일을 생성하거나 수정한다.
4. 개별 웹앱 `index.html`을 완성한다.
5. 학습지를 함께 제작한 경우 다운로드 영역을 연결한다.
6. 과목별 목록 페이지에 링크를 추가한다.
7. 브라우저에서 조작과 콘솔 오류를 확인한다.
8. 수정한 파일 목록과 확인해야 할 점을 요약한다.
