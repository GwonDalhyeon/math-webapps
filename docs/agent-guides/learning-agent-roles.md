# 수학 웹앱 제작 에이전트 역할 체계

이 문서는 새 수학 웹앱을 기획, 구현, 검토할 때 역할을 나누어 학습가치를 점검하기 위한 기준이다.

목표는 단순히 정적 웹앱을 완성하는 것이 아니라, 웹앱이 학습목표 달성에 실제로 도움이 되는지 확인하고, 부적절하면 작업지시서 단계부터 다시 설계하게 만드는 것이다.

한 사람이 모든 역할을 순서대로 수행해도 된다. 중요한 것은 구현 전과 구현 후에 학습가치 검토를 반드시 거치는 것이다.

## 기본 흐름

```text
아이디어
-> 학습설계/작업지시서 작성   ← 사용자와 논의
-> [방향 합의]
-> 사전 학습가치 검토          ┐
-> 통과 시 구현                │ 한 대화 안에서
-> 사후 학습효과 검토          │ 연속 완결
-> 최종 검증                   ┘
```

방향이 합의된 이후에는 사전 검토 → 구현 → 사후 검토 → 최종 검증을 한 대화 안에서 연속으로 완료한다. 각 단계마다 사용자 확인을 기다리지 않는다.

대화를 중단하는 경우는 다음 두 가지뿐이다.

- 사전 검토에서 `재설계` 또는 `제작 보류` 판정이 나온 경우 → 결과를 알리고 학습설계/작업지시서 단계로 돌아간다.
- 사후 검토에서 `재설계` 판정이 나온 경우 → 결과를 알리고 작업지시서 단계로 돌아간다.

`부분 수정` 판정은 스스로 수정한 뒤 같은 대화 안에서 다시 검토하고 계속 진행한다.

## 역할 1. 학습설계/작업지시서 에이전트

학습설계/작업지시서 에이전트는 만들고 싶은 웹앱 아이디어를 수업에서 쓸 수 있는 작업지시서로 바꾼다.

### 핵심 책임

- 학습목표를 고등학생 수준의 관찰 가능한 목표로 정리한다.
- 왜 웹앱이 필요한지 판단한다.
- 판서, 활동지, 교구만으로 설명하기 어려운 지점을 찾는다.
- 학생이 직접 조작해야 하는 값과 행동을 정한다.
- 조작에 따라 즉시 변해야 하는 화면 요소를 정한다.
- 학생이 관찰해야 할 변화, 누적, 수렴, 비교, 분포, 구조를 설계한다.
- 아하포인트를 한 문장으로 명시한다.
- 예상 오개념과 이를 줄이는 장치를 포함한다.
- 구현자가 바로 만들 수 있도록 파일, 화면, 계산식, 완료 기준, 검증 방법을 구체화한다.

### 반드시 답해야 할 질문

- 이 학습목표는 웹앱으로 다루면 무엇이 더 잘 보이는가?
- 학생은 무엇을 만지고, 무엇을 보며, 무엇을 말하게 되는가?
- 결과 숫자가 아니라 변화 과정이 드러나는가?
- 교사는 이 웹앱을 수업 중 어느 장면에서 몇 분 정도 사용할 수 있는가?
- 웹앱이 없어도 같은 학습효과를 더 쉽게 얻을 수 있는가?
- 학생이 흔히 오해할 지점은 무엇이며, 화면이 그 오해를 어떻게 줄이는가?

### 작업지시서에 포함할 항목

기존 `task-spec.md` 템플릿에 더해 다음 항목을 포함한다.

```text
웹앱이 필요한 이유:
판서/활동지만으로 부족한 점:
교구로 구성하기 어려운 점:
학생의 핵심 조작:
조작에 따라 즉시 변해야 하는 것:
학생이 관찰해야 할 변화:
아하포인트:
예상 오개념:
오개념을 줄이는 장치:
수업 사용 장면:
웹앱이 없어도 가능한가:
웹앱이 있을 때 더 나아지는 점:
사전 학습가치 검토 기준:
사후 학습효과 검토 기준:
```

## 역할 2. 학습가치 검토 에이전트

학습가치 검토 에이전트는 구현 전 작업지시서와 구현 후 완성된 웹앱을 모두 검토한다.

검토자는 우호적이되 냉정해야 한다. "만들 수 있는가"보다 "수업에서 왜 써야 하는가"를 우선 묻는다.

### 사전 검토

구현 전에 작업지시서를 읽고 다음을 판정한다.

- 학습목표와 핵심 조작이 직접 연결되는가?
- 학생이 조작하면서 수학적 변화 과정을 볼 수 있는가?
- 결과보다 과정, 누적, 비교, 수렴, 분포, 구조가 드러나는가?
- 아하포인트가 명확한가?
- 교사가 말이나 판서로 설명하기 어려운 부분을 대신 보여주는가?
- 교구 준비를 대체하거나 보완할 가치가 있는가?
- 단순 계산기나 보기 좋은 애니메이션에 그칠 위험은 없는가?
- 예상 오개념과 이를 줄이는 장치가 있는가?
- 수업 중 5~10분 시연 또는 학생 탐구 활동으로 쓸 수 있는가?

### 사후 검토

구현 후 웹앱을 직접 열어 조작하거나 코드를 확인하며 다음을 검토한다.

- 작업지시서의 학습목표가 화면 흐름 안에서 실제로 구현되었는가?
- 핵심 조작을 하면 시각화, 수치, 해석 문구가 즉시 함께 변하는가?
- 학생이 "왜 그런지"를 볼 수 있는 장면이 있는가?
- 화면이 너무 복잡하거나 장식적이어서 개념 집중을 방해하지 않는가?
- 버튼, 슬라이더, 초기화, 자동 실행이 수업 중 안정적으로 동작하는가?
- 오개념을 줄이기 위한 설명, 라벨, 기준선, 비교 장치가 실제로 보이는가?
- 웹앱 사용 자체가 목적이 되어 버리지 않았는가?

### 판정

검토 결과는 다음 중 하나로 명확히 남긴다.

```text
통과:
학습목표, 핵심 조작, 시각화, 수업 활용성이 잘 맞아 구현 또는 배포 가능하다.

부분 수정:
핵심 방향은 맞지만 화면 흐름, 문구, 조작 방식, 시각화, 검증 기준 중 일부 보완이 필요하다.

재설계:
웹앱의 핵심 상호작용이 학습목표와 어긋나거나 아하포인트가 약해 작업지시서부터 다시 잡아야 한다.

제작 보류:
웹앱보다 판서, 활동지, 교구, 토론, 기존 자료가 더 적합하다.
```

`통과`가 아니면 반드시 수정 지시를 함께 쓴다. 수정 지시는 "더 좋게"처럼 추상적으로 쓰지 말고, 어떤 조작, 어떤 시각화, 어떤 해석 문구를 바꿀지 작업지시서 수준으로 작성한다.

## 역할 3. 구현 에이전트

구현 에이전트는 사전 학습가치 검토를 통과한 작업지시서를 바탕으로 정적 HTML/CSS/JavaScript 웹앱을 만든다.

### 핵심 책임

- `AGENTS.md`와 `docs/agent-guides/` 지침을 따른다.
- GitHub Pages에서 바로 열리는 정적 파일 구조를 유지한다.
- React, Vite, Next.js, TypeScript, 번들러, 서버 기능을 도입하지 않는다.
- 학습목표와 직접 연결되는 핵심 조작을 우선 구현한다.
- 수학 계산 로직, 상태 관리, 화면 갱신 로직을 가능한 한 분리한다.
- 사용자의 조작에 따라 시각화, 수치, 해석 문구가 함께 반응하게 만든다.
- 결과만 출력하지 않고 변화 과정, 비교, 누적, 수렴, 분포 형성을 보여준다.
- 구현 후 학습가치 검토자가 다시 평가할 수 있도록 변경 사항과 확인 방법을 정리한다.

### 구현 후 남길 정보

최종 응답이나 작업 기록에 다음을 포함한다.

```text
수정한 파일:
구현한 핵심 조작:
조작에 따라 변하는 요소:
아하포인트가 드러나는 화면 장면:
검증한 내용:
사후 학습효과 검토가 필요한 지점:
```

## 반복 규칙

- 사전 검토 `통과` 없이는 구현하지 않는다.
- `부분 수정`은 작업지시서 또는 구현물을 고친 뒤 다시 검토한다.
- `재설계`는 기존 구현을 억지로 고치기보다 학습목표, 핵심 조작, 아하포인트를 다시 잡는다.
- `제작 보류`는 실패가 아니라 더 적절한 수업 도구를 선택한 것이다.
- 구현 편의 때문에 학습목표를 약하게 바꾸지 않는다.
- 화려한 효과보다 학생이 관찰해야 할 변화가 선명한지 우선한다.

## 간단 검토 양식

작업이 작을 때는 다음 양식을 사용해도 된다.

```md
## 학습가치 검토

판정:

학습목표 연결:

핵심 조작:

관찰할 변화:

아하포인트:

교사 설명/교구 대체 가치:

오개념 방지:

수정 지시:
```
