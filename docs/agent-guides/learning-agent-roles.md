# 수학 웹앱 제작 에이전트 역할 체계

이 문서는 새 수학 웹앱을 기획, 구현, 검토할 때 역할을 나누어 학습가치를 점검하기 위한 기준이다.

목표는 단순히 정적 웹앱을 완성하는 것이 아니라, 웹앱이 학습목표 달성에 실제로 도움이 되는지 확인하고, 부적절하면 작업지시서 단계부터 다시 설계하게 만드는 것이다.

한 사람이 모든 역할을 순서대로 수행해도 된다. 중요한 것은 구현 전과 구현 후에 학습가치 검토를 반드시 거치는 것이다.

## 기본 흐름

```text
아이디어
-> 학습설계/작업지시서 작성   ← 사용자와 논의
-> [방향 합의]
-> 사전 학습가치 검토          ┐
-> 사전 수식 표현 검토         │
-> 통과 시 구현                │ 한 대화 안에서
-> 사후 수식 표현 검토         │ 연속 완결
-> 사후 학습효과 검토          │
-> 최종 검증                   ┘
```

방향이 합의된 이후에는 사전 학습가치 검토 → 사전 수식 표현 검토 → 구현 → 사후 수식 표현 검토 → 사후 학습효과 검토 → 최종 검증을 한 대화 안에서 연속으로 완료한다. 각 단계마다 사용자 확인을 기다리지 않는다.

## 토큰 절약형 기본 모드

이 문서의 역할 체계는 품질 점검을 위한 내부 사고 절차이다. 사용자가 상세한 작업지시서나 검토 보고서를 요청하지 않는 한, 각 역할의 판단을 장문으로 출력하지 않는다.

- 학습설계/작업지시서 에이전트는 내부적으로 설계하되, 출력은 `목표`, `조작`, `변화`, `오개념 방지`, `파일` 5항목 이내로 제한한다.
- 학습가치 검토, 수학 표현 검토, 학습효과 검토는 `통과`, `부분 수정 후 통과 가능`, `부분 수정`, `재설계`, `제작 보류` 중 하나와 핵심 이유 2~4줄만 남긴다.
- 전체 작업지시서 전문은 사용자가 요청하거나, 복잡한 수업 자료를 설계하거나, 다른 도구/에이전트에 전달해야 할 때만 작성한다.
- `부분 수정` 판정은 가능한 범위에서 스스로 수정한 뒤 같은 대화 안에서 다시 확인한다.
- 최종 응답에는 수정 파일, 검증한 동작, 남은 확인점만 간단히 남긴다.

압축 양식:

```md
설계 요약:
- 목표:
- 조작:
- 변화:
- 오개념 방지:
- 파일:

검토 판정:
- 학습가치:
- 수식 표현:
- 화면 배치:
```

대화를 중단하는 경우는 다음 경우뿐이다.

- 사전 검토에서 `재설계` 또는 `제작 보류` 판정이 나온 경우 → 결과를 알리고 학습설계/작업지시서 단계로 돌아간다.
- 사전 수식 표현 검토에서 핵심 수식 표현 방식이 부적절한 경우 → 작업지시서의 수식 표현 방식부터 보완한다.
- 사후 수식 표현 검토 또는 사후 학습효과 검토에서 `재설계` 판정이 나온 경우 → 결과를 알리고 작업지시서 단계로 돌아간다.

`부분 수정` 판정은 스스로 수정한 뒤 같은 대화 안에서 다시 검토하고 계속 진행한다.

## 역할 1. 학습설계/작업지시서 에이전트

학습설계/작업지시서 에이전트는 만들고 싶은 웹앱 아이디어를 수업에서 쓸 수 있는 작업지시서로 바꾼다.

### 핵심 책임

- 학습목표를 고등학생 수준의 관찰 가능한 목표로 정리한다.
- 왜 웹앱이 필요한지 판단한다.
- 판서, 활동지, 교구만으로 설명하기 어려운 지점을 찾는다.
- 학생이 직접 조작해야 하는 값과 행동을 정한다.
- 조작에 따라 즉시 변해야 하는 화면 요소를 정한다.
- 학생이 관찰해야 할 변화, 누적, 수렴, 비교, 분포, 구조를 설계한다.
- 아하포인트를 한 문장으로 명시한다.
- 예상 오개념과 이를 줄이는 장치를 포함한다.
- 구현자가 바로 만들 수 있도록 파일, 화면, 계산식, 완료 기준, 검증 방법을 구체화한다.

### 반드시 답해야 할 질문

- 이 학습목표는 웹앱으로 다루면 무엇이 더 잘 보이는가?
- 학생은 무엇을 만지고, 무엇을 보며, 무엇을 말하게 되는가?
- 결과 숫자가 아니라 변화 과정이 드러나는가?
- 교사는 이 웹앱을 수업 중 어느 장면에서 몇 분 정도 사용할 수 있는가?
- 웹앱이 없어도 같은 학습효과를 더 쉽게 얻을 수 있는가?
- 학생이 흔히 오해할 지점은 무엇이며, 화면이 그 오해를 어떻게 줄이는가?

### 상세 작업지시서에 포함할 항목

사용자가 상세 작업지시서를 요청하거나 복잡한 웹앱을 설계할 때는 기존 `task-spec.md` 템플릿에 더해 다음 항목을 포함한다.

```text
웹앱이 필요한 이유:
판서/활동지만으로 부족한 점:
교구로 구성하기 어려운 점:
학생의 핵심 조작:
조작에 따라 즉시 변해야 하는 것:
학생이 관찰해야 할 변화:
아하포인트:
예상 오개념:
오개념을 줄이는 장치:
수업 사용 장면:
웹앱이 없어도 가능한가:
웹앱이 있을 때 더 나아지는 점:
사전 학습가치 검토 기준:
사전 수식 표현 검토 기준:
수업 화면/전자칠판 검토 기준:
사후 학습효과 검토 기준:
사후 수식 표현 검토 기준:
```

## 역할 2. 학습가치 검토 에이전트

학습가치 검토 에이전트는 구현 전 작업지시서와 구현 후 완성된 웹앱을 모두 검토한다.

검토자는 우호적이되 냉정해야 한다. "만들 수 있는가"보다 "수업에서 왜 써야 하는가"를 우선 묻는다.

### 사전 검토

구현 전에 작업지시서를 읽고 다음을 판정한다.

- 학습목표와 핵심 조작이 직접 연결되는가?
- 학생이 조작하면서 수학적 변화 과정을 볼 수 있는가?
- 결과보다 과정, 누적, 비교, 수렴, 분포, 구조가 드러나는가?
- 아하포인트가 명확한가?
- 교사가 말이나 판서로 설명하기 어려운 부분을 대신 보여주는가?
- 교구 준비를 대체하거나 보완할 가치가 있는가?
- 단순 계산기나 보기 좋은 애니메이션에 그칠 위험은 없는가?
- 예상 오개념과 이를 줄이는 장치가 있는가?
- 수업 중 5~10분 시연 또는 학생 탐구 활동으로 쓸 수 있는가?

## 역할 3. 수학 표현 검토 에이전트

수학 표현 검토 에이전트는 구현 전 작업지시서와 구현 후 웹앱을 대상으로 수식의 정확성, 렌더링 방식, 학생 가독성을 점검한다.

수식 표현은 계산식이 맞는지만 보지 않는다. 교과서와 전자칠판에서 학생이 읽는 방식으로 표현되는지 확인한다.

### 핵심 책임

- 복잡한 수식이 LaTeX + MathJax 또는 MathML처럼 구조가 보이는 방식으로 표시되는지 확인한다.
- 분수는 `1/2`, `(X-np)/σ`, `a/b` 같은 슬래시 텍스트만으로 두지 않고, 교과서처럼 세로 분수 형태로 렌더링되는지 확인한다.
- 제곱근, 합, 적분, 극한, 확률식, 조건부확률, 표준화 식처럼 구조가 중요한 수식은 텍스트식보다 렌더링 수식을 우선하게 한다.
- 화면에 원시 LaTeX 코드가 그대로 노출되지 않는지 확인한다.
- 동적으로 바뀌는 수식이 조작 후 다시 렌더링되는지 확인한다.
- 변수와 기호가 조작값, 그래프, 표, 해석 문구에서 일관되는지 확인한다.
- 학생이 수식을 읽는 데 필요한 짧은 설명과 그래프 연결이 있는지 확인한다.

### 사전 수식 표현 검토

구현 전에 작업지시서를 읽고 다음을 판정한다.

- 어떤 수식을 화면에 직접 보여줄지 정해져 있는가?
- 분수, 제곱근, 확률식, 표준화 식처럼 구조가 중요한 수식은 LaTeX 또는 MathML 렌더링 대상으로 지정되어 있는가?
- 슬래시 텍스트로 쓰면 오해될 수 있는 수식이 없는가?
- 수식 렌더링에 MathJax CDN을 쓰는 경우 이유와 오프라인 제약이 적혀 있는가?
- 수식이 그래프, 도식, 현재 상태 문구와 연결되도록 설계되어 있는가?

### 사후 수식 표현 검토

구현 후 웹앱을 직접 열어 보거나 코드를 확인하며 다음을 검토한다.

- 분수가 교과서식 세로 분수로 보이는가?
- 원시 LaTeX 문자열이 화면에 그대로 남아 있지 않은가?
- 슬라이더나 버튼 조작 뒤 동적 수식이 최신 값으로 갱신되고 다시 렌더링되는가?
- 변수 `n`, `p`, `X`, `μ`, `σ`, `Z` 같은 기호가 한 화면 안에서 같은 의미로 쓰이는가?
- 수식과 그래프가 같은 평균, 표준편차, 확률값, 경계값을 가리키는가?
- 수식이 너무 작거나 잘려서 프로젝터/전자칠판에서 읽기 어려운 부분은 없는가?

### 판정

수식 표현 검토 결과는 다음 중 하나로 남긴다.

```text
통과:
수식의 구조, 렌더링, 기호 일관성, 학생 가독성이 충분하다.

부분 수정:
수식 방향은 맞지만 분수 렌더링, 동적 재렌더링, 기호 설명, 크기나 배치 일부 보완이 필요하다.

재설계:
수식 표현 방식이 학습목표를 방해하거나, 핵심 수식이 구조적으로 보이지 않아 작업지시서부터 다시 잡아야 한다.
```

`부분 수정`이면 어떤 수식을 어떤 방식으로 바꿀지 구체적으로 지시한다. 예를 들어 "`(X-np)/σ` 텍스트를 `\frac{X-np}{\sigma}`로 렌더링하고, 조작 후 MathJax 재렌더링을 호출한다"처럼 쓴다.

## 역할 4. 구현 에이전트

구현 에이전트는 사전 학습가치 검토와 사전 수식 표현 검토를 통과한 작업지시서를 바탕으로 정적 HTML/CSS/JavaScript 웹앱을 만든다.

### 핵심 책임

- `AGENTS.md`와 `docs/agent-guides/` 지침을 따른다.
- GitHub Pages에서 바로 열리는 정적 파일 구조를 유지한다.
- React, Vite, Next.js, TypeScript, 번들러, 서버 기능을 도입하지 않는다.
- 학습목표와 직접 연결되는 핵심 조작을 우선 구현한다.
- 수학 계산 로직, 상태 관리, 화면 갱신 로직을 가능한 한 분리한다.
- 사용자의 조작에 따라 시각화, 수치, 해석 문구가 함께 반응하게 만든다.
- 결과만 출력하지 않고 변화 과정, 비교, 누적, 수렴, 분포 형성을 보여준다.
- 수식은 필요한 경우 LaTeX + MathJax 또는 MathML로 렌더링하고, 동적 수식은 조작 후 다시 렌더링되게 한다.
- 구현 후 학습가치 검토자와 수학 표현 검토자가 다시 평가할 수 있도록 변경 사항과 확인 방법을 정리한다.

## 역할 5. 학습효과 검토 에이전트

학습효과 검토 에이전트는 구현 후 완성된 웹앱이 수업 장면에서 실제로 쓰기 좋은지 검토한다.

검토자는 학습가치뿐 아니라 교사가 프로젝터나 전자칠판으로 보여줄 때의 화면 구성을 함께 본다.

### 사후 검토

구현 후 웹앱을 직접 열어 조작하거나 코드를 확인하며 다음을 검토한다.

- 작업지시서의 학습목표가 화면 흐름 안에서 실제로 구현되었는가?
- 핵심 조작을 하면 시각화, 수치, 해석 문구가 즉시 함께 변하는가?
- 학생이 "왜 그런지"를 볼 수 있는 장면이 있는가?
- 화면이 너무 복잡하거나 장식적이어서 개념 집중을 방해하지 않는가?
- 교사가 프로젝터나 전자칠판으로 보여줄 때 조작 영역과 핵심 결과 시각화가 한 화면 안에 함께 보이는가?
- 핵심 슬라이더, 버튼, 그래프, 현재 결과, 해석 문구가 첫 화면 또는 아주 짧은 스크롤 안에 들어오는가?
- 수업 시연 중 그래프를 보려고 조작 영역을 잃거나, 조작하려고 결과 화면을 잃는 배치가 아닌가?
- 교실 PC/전자칠판 화면에서 글자와 수식이 충분히 크고 버튼을 누르기 쉬운가?
- 모바일 대응 때문에 교실 시연 화면의 밀도와 가독성이 희생되지 않았는가?
- 버튼, 슬라이더, 초기화, 자동 실행이 수업 중 안정적으로 동작하는가?
- 오개념을 줄이기 위한 설명, 라벨, 기준선, 비교 장치가 실제로 보이는가?
- 웹앱 사용 자체가 목적이 되어 버리지 않았는가?

### 판정

검토 결과는 다음 중 하나로 명확히 남긴다.

```text
통과:
학습목표, 핵심 조작, 시각화, 수업 활용성이 잘 맞아 구현 또는 배포 가능하다.

부분 수정:
핵심 방향은 맞지만 화면 흐름, 문구, 조작 방식, 시각화, 검증 기준 중 일부 보완이 필요하다.

재설계:
웹앱의 핵심 상호작용이 학습목표와 어긋나거나 아하포인트가 약해 작업지시서부터 다시 잡아야 한다.

제작 보류:
웹앱보다 판서, 활동지, 교구, 토론, 기존 자료가 더 적합하다.
```

`통과`가 아니면 반드시 수정 지시를 함께 쓴다. 수정 지시는 "더 좋게"처럼 추상적으로 쓰지 말고, 어떤 조작, 어떤 시각화, 어떤 해석 문구를 바꿀지 작업지시서 수준으로 작성한다.

### 구현 후 남길 정보

최종 응답이나 작업 기록에 다음을 포함한다.

```text
수정한 파일:
구현한 핵심 조작:
조작에 따라 변하는 요소:
아하포인트가 드러나는 화면 장면:
검증한 내용:
수식 표현 검토 결과:
수업 화면/전자칠판 검토 결과:
사후 학습효과 검토가 필요한 지점:
```

## 반복 규칙

- 사전 검토 `통과` 없이는 구현하지 않는다.
- 사전 수식 표현 검토 `통과` 없이는 구현하지 않는다.
- `부분 수정`은 작업지시서 또는 구현물을 고친 뒤 다시 검토한다.
- `재설계`는 기존 구현을 억지로 고치기보다 학습목표, 핵심 조작, 아하포인트를 다시 잡는다.
- `제작 보류`는 실패가 아니라 더 적절한 수업 도구를 선택한 것이다.
- 구현 편의 때문에 학습목표를 약하게 바꾸지 않는다.
- 구현 편의 때문에 구조가 중요한 수식을 슬래시 텍스트로만 남기지 않는다.
- 화려한 효과보다 학생이 관찰해야 할 변화가 선명한지 우선한다.

## 간단 검토 양식

작업이 작거나 일반적인 단일 웹앱 제작일 때는 다음 압축 양식을 우선 사용한다.

```md
설계 요약:
- 목표:
- 조작:
- 변화:
- 오개념 방지:
- 파일:

검토 판정:
- 학습가치:
- 수식 표현:
- 화면 배치:
- 수정 지시:
```
