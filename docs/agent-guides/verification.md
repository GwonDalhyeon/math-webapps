# 완료 전 검증 지침

이 문서는 웹앱, 학습지, PDF, 작업지시서 작업을 마치기 전에 읽는다.

## 공통 검증

모든 작업 후 확인한다.

- 수정한 파일이 사용자의 요청과 직접 관련이 있는가
- 의도하지 않은 기존 변경을 되돌리지 않았는가
- 새 파일 이름과 경로가 영어 소문자, 숫자, 하이픈 규칙을 따르는가
- 공개 저장소에 올라가면 안 되는 정보가 포함되지 않았는가
- 최종 응답에 수정한 파일과 확인한 내용을 간단히 요약했는가

## 웹앱 검증

새 웹앱 또는 웹앱 수정 후 확인한다.

- 화면 검토의 최우선 통과 기준은 iPad A16이다. 가로 `1180×820` CSS px에서 핵심 조작, 그래프 또는 도식, 현재 결과, 짧은 해석 문구가 첫 화면 또는 아주 짧은 스크롤 안에 연결되어야 한다.
- iPad A16 세로 `820×1180` CSS px에서도 주요 버튼, 슬라이더, 수식, 그래프, 결과 카드가 겹치거나 잘리지 않아야 한다.
- iPad A16 기준에서 실패하면 프로젝터/전자칠판이나 데스크톱에서 통과해도 화면 배치 판정은 `부분 수정` 이상으로 남긴다.
- GitHub Pages에서 실행 가능한 정적 파일 구조인가
- 로컬 절대경로가 없는가
- 공통 CSS 상대경로가 올바른가
- 모든 링크가 실제 파일 또는 폴더를 가리키는가
- 학습지를 함께 제작하거나 탑재하기로 한 경우, 공개 범위에 맞는 Markdown/PDF만 웹앱 폴더에 있고 다운로드 링크가 실제 파일로 연결되어 있는가
- 다운로드 자료가 3개 이상이면 접이식 영역으로 묶여 화면을 과하게 차지하지 않는가
- 실제 파일이 없는 준비 중 항목은 링크가 아니라 `article.card`와 `.status`로 표시했는가
- 브라우저 콘솔 오류가 없는가
- 버튼, 슬라이더, 초기화, 자동 실행이 동작하는가
- 자동 실행 중 중복 타이머가 생기지 않는가
- 교실 프로젝터/전자칠판 화면에서도 조작 영역과 핵심 결과 시각화가 한 화면 안에 함께 보이는가
- 핵심 슬라이더, 버튼, 그래프 또는 도식, 현재 결과, 짧은 해석 문구가 첫 화면 또는 아주 짧은 스크롤 안에 들어오는가
- 수업 시연 중 조작 영역과 결과 영역을 오가느라 과도한 스크롤이 필요하지 않은가
- 모바일과 교실 프로젝터 화면에서 주요 요소가 깨지지 않는가
- 새 웹앱이 필요한 과목별 `index.html`에 등록되었는가

## 수학 검증

수학 웹앱과 학습지에서 확인한다.

- 사전 학습가치 검토를 통과한 작업지시서에 따라 구현했는가
- 이 앱이 보여주는 수학적 변화가 분명한가
- 사용자가 조작하면 어떤 요소가 즉시 바뀌는가
- 그래프나 도식이 왜 필요한지 설명 가능한가
- 학생이 현재 상태를 한눈에 읽을 수 있는가
- 결과, 과정, 해석이 같은 화면 또는 학습 흐름 안에서 연결되는가
- 변수와 기호가 한 웹앱 안에서 일관되는가
- 분수, 제곱근, 합, 적분, 극한, 확률식, 표준화 식처럼 구조가 중요한 수식이 LaTeX + MathJax 또는 MathML로 렌더링되는가
- 분수가 `1/2`, `(X-np)/σ`, `a/b` 같은 슬래시 텍스트만으로 남지 않고 교과서식 세로 분수로 보이는가
- 화면에 원시 LaTeX 코드가 그대로 노출되지 않는가
- 동적으로 바뀌는 수식이 조작 후 최신 값으로 갱신되고 다시 렌더링되는가
- 수식과 그래프가 같은 평균, 표준편차, 확률값, 경계값을 가리키는가
- 수식 크기와 줄바꿈이 프로젝터/전자칠판에서 읽기 좋은가
- 표본 크기와 반복 횟수처럼 헷갈리기 쉬운 개념을 구분했는가
- 아하포인트가 실제 화면 흐름 안에 드러나는가
- 교사가 판서나 교구로 구성하기 어려운 부분을 보완하는가
- 웹앱 사용 자체가 목적이 되지 않고 학습목표 달성을 돕는가
- 구현 후 사후 수식 표현 검토 판정을 남겼는가
- 구현 후 사후 학습효과 검토 판정을 남겼는가
- 사후 검토 판정은 장문 보고서가 아니라 핵심 이유 2~4줄의 압축 결과로 남겼는가

## 학습지 검증

학습지 Markdown 작성 후 확인한다.

- 학습지를 새로 제작하거나 웹앱에 탑재하기로 했다면 해당 웹앱 폴더 안에 있는가
- 웹앱에 맞는 학습지를 공개하기로 했다면 웹앱 `index.html`에서 다운로드할 수 있는가
- 요청 범위에 학생용과 교사용이 모두 포함된 경우 두 자료가 함께 있는가
- 학생용과 교사용을 모두 관리하는 작업에서 학생용 학습지를 수정했다면 대응하는 교사용 학습지도 함께 갱신했는가
- 교사용 학습지를 제작하거나 수정하는 작업이라면 예시답안, 계산 과정, 지도 포인트가 필요한 만큼 포함되어 있는가
- 모든 수식이 LaTeX 형식인가
- 인라인 수식은 `$...$`, 별도 줄 수식은 `$$...$$`인가
- 웹앱 기본값과 학습지 계산값이 일치하는가
- 학생이 직접 계산할 수 있는 값인가
- 관찰 기록 칸과 해석 문항이 충분한가
- 마지막에 개념 정리 문항이 있는가

## PDF 검증

PDF 변환 후 확인한다.

- 목표 페이지 수가 맞는가
- 요청 범위에 맞는 PDF가 만들어졌는가
- 학생용 PDF와 교사용 PDF를 모두 만들기로 한 경우 두 파일이 함께 있는가
- 교사용 PDF를 함께 만드는 작업에서 페이지 수가 학생용과 다를 경우 그 이유를 설명했는가
- 한글이 깨지지 않는가
- 수식이 읽기 좋게 표시되는가
- 여백이 과하지 않은가
- 표와 빈칸이 잘리지 않는가
- Markdown 원본을 수정했는지, PDF만 다시 만들었는지 구분했는가

## 작업지시서 검증

작업지시서 작성 후 확인한다.

- 위치가 `tasks/`인가
- 파일명이 `YYYY-MM-DD-topic-name.md` 형식인가
- 웹앱 이름, 위치, 대상 학생, 학습 목표가 있는가
- 웹앱이 필요한 이유와 판서/활동지만으로 부족한 점이 있는가
- 학생의 핵심 조작, 관찰할 변화, 아하포인트가 구체적인가
- 예상 오개념과 이를 줄이는 장치가 있는가
- 사전 학습가치 검토 기준과 사후 학습효과 검토 기준이 있는가
- 사용자 조작, 시각화 요소, 상태 표시가 구체적인가
- 수식 표현 방식에 교과서식 분수 렌더링, MathJax/MathML 사용 여부, 동적 재렌더링 기준이 있는가
- 수업 화면/전자칠판 배치 기준에 한 화면 안에서 조작과 결과를 함께 볼 수 있다는 조건이 있는가
- 화면 검토 기준에 iPad A16 가로 `1180×820` CSS px 우선 검증과 세로 `820×1180` CSS px 보조 검증이 들어 있는가
- 생성할 파일과 수정할 파일이 명확한가
- 수정하지 말 것이 적혀 있는가
- 완료 기준과 검증 방법이 실행 가능한 체크리스트인가

## 최종 응답

최종 응답에는 필요한 만큼만 적는다.

- 무엇을 만들었거나 수정했는지
- 주요 파일 경로
- 검증한 내용
- 확인하지 못한 점이 있으면 이유
- 상세 작업지시서나 긴 역할별 검토 보고서를 요청받지 않았다면 압축 요약만 제공했는지

파일 경로는 가능한 한 클릭 가능한 절대 경로로 제공한다.
