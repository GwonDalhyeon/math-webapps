# math-webapps Claude Code 지침

이 프로젝트는 고등학교 수학 개념을 시각화하고 탐구할 수 있는 수업용 웹앱 모음이다.  
주요 사용자는 일반계 고등학교 평균 수준 학생과 수업 시연을 하는 교사이다.

핵심 목표는 학생이 수학을 눈으로 보고 직접 조작하면서, 결과뿐 아니라 변화 과정, 누적, 수렴, 분포 형성을 직관적으로 이해하도록 돕는 것이다.

---

## 1. 프로젝트 기본 방향

- 이 프로젝트는 GitHub Pages로 배포되는 정적 웹사이트이다.
- 기본 기술은 HTML, CSS, JavaScript이다.
- 설치나 빌드 없이 실행되는 단일 HTML 웹앱을 우선한다.
- React, Vite, Next.js, TypeScript, 번들러는 사용자가 명시적으로 요청하지 않는 한 사용하지 않는다.
- 서버 기능, 데이터베이스, 로그인, API 서버, 서버리스 함수는 사용하지 않는다.
- 모든 결과물은 GitHub Pages에서 바로 열려야 한다.
- 티스토리는 웹앱 실행 공간이 아니라 설명 글과 실행 버튼을 제공하는 공간으로 본다.

## 2. 권장 폴더 구조

프로젝트의 기본 구조는 다음을 따른다.

```text
math-webapps/
├─ index.html
├─ assets/
│  ├─ common.css
│  └─ common.js
│
├─ probability-statistics/
│  ├─ index.html
│  ├─ counting/
│  │  └─ index.html
│  ├─ monty-hall/
│  │  └─ index.html
│  ├─ conditional-probability/
│  │  └─ index.html
│  ├─ binomial-distribution/
│  │  └─ index.html
│  └─ normal-distribution/
│     └─ index.html
│
└─ calculus/
   ├─ index.html
   ├─ sequence-limit/
   │  └─ index.html
   ├─ infinite-series/
   │  └─ index.html
   ├─ derivative-graph/
   │  └─ index.html
   ├─ integral-area/
   │  └─ index.html
   └─ definite-integral/
      └─ index.html
```

역할은 다음과 같다.

- `/index.html`: 전체 수학 웹앱 자료실 첫 화면
- `/probability-statistics/index.html`: 확률과 통계 웹앱 목록
- `/calculus/index.html`: 미적분 웹앱 목록
- 각 하위 폴더의 `index.html`: 실제 실행되는 개별 웹앱

## 3. 파일명과 경로 규칙

- 폴더명과 파일명은 영어 소문자, 숫자, 하이픈만 사용한다.
- 한글, 공백, 특수문자는 URL 깨짐을 막기 위해 사용하지 않는다.
- 각 웹앱은 반드시 자기 폴더 안의 `index.html`로 실행되게 한다.
- 상대경로를 정확히 작성한다.

예:

```html
<link rel="stylesheet" href="../../assets/common.css">
```

루트의 `index.html`에서 과목 폴더로 연결할 때는 다음처럼 작성한다.

```html
<a href="./probability-statistics/">확률과 통계</a>
<a href="./calculus/">미적분</a>
```

과목별 목록 페이지에서 개별 웹앱으로 연결할 때는 다음처럼 작성한다.

```html
<a href="./conditional-probability/">조건부확률</a>
```

## 4. 공통 디자인 원칙

공통 스타일은 가능한 한 `/assets/common.css`에 둔다.

디자인은 다음을 우선한다.

- 교실 프로젝터에서 잘 보이는 큰 글씨
- 높은 명도 대비
- 단순한 카드형 구조
- 불필요하게 화려하지 않은 UI
- 조작 영역과 시각화 영역의 명확한 분리
- 모바일에서도 기본적으로 깨지지 않는 반응형 레이아웃

단, 개별 웹앱에만 필요한 스타일은 해당 웹앱의 `index.html` 안에 `<style>`로 포함해도 된다.

## 5. 개별 웹앱 필수 구성

각 웹앱은 가능하면 다음 흐름을 가진다.

```text
설명 → 조작 → 변화 확인 → 해석
```

화면에는 가능한 한 다음 요소를 포함한다.

1. 짧은 개념 설명
2. 핵심 시각화 영역
3. 사용자 조작 영역
4. 실행 버튼
5. 현재 상태 또는 결과 해석 영역

권장 버튼 체계는 다음과 같다.

- 1회 실행
- 단계 실행
- 10회 실행
- 100회 실행
- 자동 실행
- 일시정지
- 초기화

개념에 따라 모든 버튼이 필요하지 않다면 생략해도 되지만, 반복 실험·누적·분포 형성 앱에서는 가능한 한 포함한다.

## 6. 수학 웹앱 설계 원칙

웹앱은 단순 계산기가 아니라 수학적 변화 과정을 보여주는 도구여야 한다.

다음을 항상 고려한다.

- 사용자가 값을 바꾸면 무엇이 즉시 변하는가?
- 그래프, 도식, 표, 수치, 설명 문구가 함께 반응하는가?
- 학생이 현재 상태를 한눈에 읽을 수 있는가?
- 최종 결과만 보여주지 않고 중간 과정, 누적 변화, 수렴 또는 분포 형성을 보여주는가?
- 조작 결과가 수학 개념과 직접 연결되는가?
- 교사가 수업 중 짧게 시연하기에 충분히 단순한가?

피해야 할 것:

- 결과 숫자만 출력하는 앱
- 인터랙션이 거의 없는 앱
- 의미가 불분명한 그래프
- 한 화면에 너무 많은 요소를 넣는 구성
- 보기에는 화려하지만 수학적 의미가 약한 효과
- 학생이 같은 작업을 지나치게 반복해야만 이해되는 구조

## 7. 우선 다룰 수학 범위

### 확률과 통계

- 경우의 수
- 확률의 뜻
- 확률의 덧셈정리
- 조건부확률
- 사건의 독립과 종속
- 확률변수와 확률분포
- 기댓값과 표준편차
- 이항분포
- 정규분포
- 모집단과 표본
- 추정

### 미적분

- 수열의 극한
- 급수
- 여러 가지 함수의 미분
- 여러 가지 미분법
- 도함수의 활용
- 여러 가지 적분법
- 정적분의 활용

계산 결과보다 개념의 의미와 과정이 보이도록 구성한다.

## 8. 코드 작성 원칙

- 전체 코드는 바로 실행 가능한 완성본으로 작성한다.
- HTML, CSS, JavaScript를 읽기 쉽게 작성한다.
- 변수명과 함수명은 의미가 드러나게 작성한다.
- 수학 계산 로직과 화면 갱신 로직을 가능하면 분리한다.
- 필요한 부분에는 짧은 주석을 단다.
- 자동 실행, 일시정지, 초기화 기능에서 상태가 꼬이지 않게 한다.
- 입력값 오류나 범위 초과를 고려한다.
- 외부 라이브러리는 꼭 필요한 경우에만 CDN으로 사용한다.
- CDN을 사용할 때는 사용 이유를 코드 주석이나 설명에 간단히 남긴다.
- 가능하면 Canvas, SVG, 기본 DOM 조작만으로 구현한다.

## 9. 접근성과 수업 사용성

다음을 지킨다.

- 버튼과 슬라이더는 충분히 크게 만든다.
- 숫자 결과는 단위나 의미를 함께 표시한다.
- 색만으로 의미를 구분하지 말고 텍스트나 모양도 함께 사용한다.
- 중요한 변화는 설명 문구로도 알려준다.
- 수업 시연용으로 처음 보는 학생도 흐름을 이해할 수 있게 한다.
- 모바일보다 교실 PC와 프로젝터 환경을 우선하되, 모바일에서도 기본 조작은 가능하게 한다.

## 10. 목록 페이지 관리 규칙

새 웹앱을 추가할 때는 반드시 다음을 함께 수정한다.

1. 해당 웹앱 폴더 생성
2. 해당 폴더 안에 `index.html` 생성
3. 과목별 `index.html` 목록에 카드 추가
4. 루트 `index.html`의 전체 현황이 필요하면 함께 수정
5. 준비 중인 웹앱은 실제 파일이 없으면 링크하지 않는다

준비 중인 항목은 다음처럼 표시한다.

```html
<span class="status">준비 중</span>
```

실제 페이지가 없는 링크는 만들지 않는다.

---

## 11. GitHub Pages 배포 기준

GitHub Pages 배포를 전제로 하므로 다음을 지킨다.

- 모든 링크는 상대경로로 작성한다.
- 로컬에서 동작하더라도 GitHub Pages에서 깨질 수 있는 절대 로컬 경로를 사용하지 않는다.
- 다음과 같은 경로는 사용하지 않는다.

```html
<link rel="stylesheet" href="C:/Users/.../style.css">
```

- 다음처럼 상대경로를 사용한다.

```html
<link rel="stylesheet" href="../../assets/common.css">
```

- 각 폴더는 `/folder-name/` 형태로 접근해도 열리도록 `index.html`을 둔다.

## 12. Claude Code 작업 방식

Claude Code는 다음 방식으로 작업한다.

- 먼저 현재 폴더 구조를 확인한다.
- 기존 파일을 수정하기 전에 역할과 연결 구조를 파악한다.
- 새 웹앱을 만들 때 전체 구조를 깨지 않는다.
- 이미 있는 공통 스타일을 최대한 재사용한다.
- 하나의 요청에서 지나치게 많은 웹앱을 한꺼번에 만들지 않는다.
- 사용자가 요청한 웹앱 하나를 완성도 있게 만드는 것을 우선한다.
- 작업 후 수정한 파일 목록과 확인해야 할 점을 요약한다.

사용자가 명시하지 않는 한 하지 말아야 할 것:

- 프로젝트를 React, Vite, Next.js로 전환
- 패키지 설치
- 빌드 과정 추가
- 서버 기능 추가
- 기존 폴더 구조 대폭 변경
- 불필요한 외부 라이브러리 추가
- 실제 파일이 없는 링크 생성

## 13. 웹앱 완성 전 자체 점검

각 웹앱을 만든 뒤 다음 기준으로 스스로 점검한다.

- 이 앱이 보여주는 수학적 변화는 무엇인가?
- 사용자가 조작하면 어떤 요소가 즉시 바뀌는가?
- 그래프나 도식은 왜 필요한가?
- 학생이 현재 상태를 한눈에 읽을 수 있는가?
- 초기화와 반복 실행이 안정적인가?
- 자동 실행 중 중복 타이머가 생기지 않는가?
- 설명 없이도 사용 흐름을 어느 정도 이해할 수 있는가?
- 교사가 수업에서 바로 사용할 만큼 가독성이 충분한가?
- 모바일 화면에서 주요 버튼과 시각화가 깨지지 않는가?
- GitHub Pages 상대경로가 올바른가?

---

## 14. 웹앱 요청을 받았을 때의 기본 작업 순서

사용자가 새 웹앱 제작을 요청하면 다음 순서로 작업한다.

1. 학습 목표를 짧게 정리한다.
2. 화면 구성을 간단히 설계한다.
3. 필요한 파일을 생성하거나 수정한다.
4. 개별 웹앱 `index.html`을 완성한다.
5. 과목별 목록 페이지에 링크를 추가한다.
6. 로컬 확인 방법과 GitHub Pages 주소 예시를 알려준다.

---

## 15. 좋은 웹앱의 기준

이 프로젝트에서 좋은 웹앱은 다음과 같다.

- 정답을 보여주는 앱이 아니라 수학이 변화하는 모습을 보여주는 앱
- 학생이 직접 조작하며 패턴을 발견할 수 있는 앱
- 교사가 수업 중 짧게 보여줘도 핵심이 전달되는 앱
- 결과, 과정, 해석이 한 화면 안에서 연결되는 앱
- 코드가 단순하고 장기적으로 유지보수하기 쉬운 앱

최종 목표는 “웹에서 바로 실행되는 수학 개념 실험실”을 만드는 것이다.
