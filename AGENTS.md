# math-webapps Codex 지침

> 이 파일(AGENTS.md)은 Codex용이다. Claude Code용 지침은 CLAUDE.md에 별도로 관리한다.  
> 공통 세부 지침은 `docs/agent-guides/`에 두며, 도구별 작업 방식처럼 역할이 다른 부분을 제외한 핵심 원칙은 두 파일에서 함께 유지한다.

이 프로젝트는 고등학교 수학 개념을 시각화하고 탐구할 수 있는 수업용 웹앱 모음이다.  
주요 사용자는 일반계 고등학교 평균 수준 학생과 수업 시연을 하는 교사이다.

핵심 목표는 학생이 수학을 눈으로 보고 직접 조작하면서, 결과뿐 아니라 변화 과정, 누적, 수렴, 분포 형성을 직관적으로 이해하도록 돕는 것이다.

## 작업별 추가 지침

상세 기준은 `docs/agent-guides/`에 나누어 둔다. 작업을 시작할 때 요청과 직접 관련된 문서만 우선 읽고 적용한다.

- 새 웹앱을 만들거나 핵심 기능을 수정할 때: `docs/agent-guides/webapp.md`
- 새 웹앱의 작업지시서 작성, 사전/사후 학습가치 검토를 할 때: `docs/agent-guides/learning-agent-roles.md`
- 학습지 Markdown을 만들거나 수정할 때: `docs/agent-guides/worksheet.md`
- 학습지를 PDF로 변환하거나 페이지 수를 맞출 때: `docs/agent-guides/pdf-export.md`
- 작업지시서를 작성할 때: `docs/agent-guides/task-spec.md`
- 폴더, 경로, 목록 페이지, GitHub Pages 링크를 수정할 때: `docs/agent-guides/repo-structure.md`
- 작업 완료 전 자체 점검을 할 때: `docs/agent-guides/verification.md`

## 토큰 절약형 에이전트 운영 원칙

웹앱 구성 에이전트는 학습가치와 수학 표현 검토를 깊게 수행하되, 사용자에게는 기본적으로 짧게 보고한다.

- 에이전트 역할 분석은 기본적으로 내부 검토로 수행하고, 장문 보고서로 출력하지 않는다.
- 새 웹앱 제작 전 설계 요약은 다음 5항목 이내로 제한한다.
  1. 학습 목표
  2. 핵심 조작
  3. 즉시 변하는 시각 요소
  4. 예상 오개념 방지 장치
  5. 구현 파일
- 사전/사후 검토 결과는 `통과`, `부분 수정 후 통과 가능`, `부분 수정`, `재설계`, `제작 보류` 중 하나와 핵심 이유 2~4줄로만 작성한다.
- 사용자가 명시적으로 요청하지 않는 한 전체 작업지시서 전문을 작성하거나 `tasks/`에 저장하지 않는다.
- 복잡한 웹앱, 여러 차시 수업 자료, 다른 도구/에이전트 전달용 작업지시서가 필요한 경우에만 상세 작업지시서를 작성한다.
- 최종 응답은 수정 파일, 확인한 동작, 남은 확인점만 간단히 요약한다.
- 필요한 문서만 읽는다. 새 웹앱이면 `webapp.md`와 `learning-agent-roles.md`, 완료 전에는 `verification.md`를 우선하고, 학습지/PDF/경로 작업 문서는 해당 작업이 있을 때만 읽는다.

압축 보고 양식:

```md
설계 요약:
- 목표:
- 조작:
- 변화:
- 오개념 방지:
- 파일:

검토 판정:
- 학습가치:
- 수식 표현:
- 화면 배치:
```

핵심 원칙은 "검토는 깊게 하되, 출력은 판정과 핵심 근거만 짧게 남긴다"이다.

## 1. 프로젝트 기본 방향

- 이 프로젝트는 GitHub Pages로 배포되는 정적 웹사이트이다.
- 기본 기술은 HTML, CSS, JavaScript이다.
- 설치나 빌드 없이 실행되는 단일 HTML 웹앱을 우선한다.
- React, Vite, Next.js, TypeScript, 번들러는 사용자가 명시적으로 요청하지 않는 한 사용하지 않는다.
- 서버 기능, 데이터베이스, 로그인, API 서버, 서버리스 함수는 사용하지 않는다.
- 모든 결과물은 GitHub Pages에서 바로 열려야 한다.
- 티스토리는 웹앱 실행 공간이 아니라 설명 글과 실행 버튼을 제공하는 공간으로 본다.

## 2. GitHub Pages 기술 스택

이 프로젝트의 실행 환경은 GitHub Pages의 정적 파일 호스팅이다. 완성된 웹앱은 사용자가 GitHub Pages URL을 열었을 때 브라우저만으로 즉시 실행되어야 한다.

사용 가능:

- HTML
- CSS
- 브라우저 JavaScript
- Canvas
- SVG
- MathML
- 정적 이미지, 오디오, JSON, CSV 파일
- 브라우저 내장 API
- 필요한 경우 CDN 기반 클라이언트 라이브러리

조건부 사용:

- 단순한 수식은 텍스트 표기 또는 MathML로 충분한지 먼저 판단한다.
- LaTeX 렌더링이 필요한 경우 MathJax CDN을 사용할 수 있다.
- MathJax 같은 수식 렌더링 CDN은 필요한 페이지에만 추가한다.
- 외부 CDN을 사용할 경우 인터넷 연결이 필요하다는 점을 고려한다.
- JavaScript 모듈을 사용할 경우 상대경로로 동작해야 한다.

사용 금지:

- Node.js 런타임을 전제로 한 실행 구조
- npm install이 필요한 구조
- 빌드 과정
- 서버 코드
- 데이터베이스
- 로그인 또는 회원 기능
- 환경변수
- 서버 API 호출을 전제로 한 기능
- 로컬 절대경로

## 4. 파일명과 경로 규칙

- 폴더명과 파일명은 영어 소문자, 숫자, 하이픈만 사용한다.
- 한글, 공백, 특수문자는 URL 깨짐을 막기 위해 사용하지 않는다.
- 각 웹앱은 반드시 자기 폴더 안의 `index.html`로 실행되게 한다.
- 상대경로를 정확히 작성한다.

예:

```html
<link rel="stylesheet" href="../../assets/common.css">
```

루트의 `index.html`에서 과목 폴더로 연결할 때는 다음처럼 작성한다.

```html
<a href="./probability-statistics/">확률과 통계</a>
<a href="./calculus/">미적분</a>
<a href="./gifted/">영재원 수업</a>
```

과목별 목록 페이지에서 개별 웹앱으로 연결할 때는 다음처럼 작성한다.

```html
<a href="./monty-hall/">몬티홀 문제</a>
```

## 5. 공통 디자인 원칙

공통 스타일은 가능한 한 `/assets/common.css`에 둔다.

디자인은 다음을 우선한다.

- 교실 프로젝터에서 잘 보이는 큰 글씨
- 높은 명도 대비
- 단순한 카드형 구조
- 불필요하게 화려하지 않은 UI
- 조작 영역과 시각화 영역의 명확한 분리
- 모바일에서도 기본적으로 깨지지 않는 반응형 레이아웃

단, 개별 웹앱에만 필요한 스타일은 해당 웹앱의 `index.html` 안에 `<style>`로 포함해도 된다.

## 6. 개별 웹앱 필수 구성

각 웹앱은 가능하면 다음 흐름을 가진다.

```text
설명 → 조작 → 변화 확인 → 해석
```

화면에는 가능한 한 다음 요소를 포함한다.

1. 짧은 개념 설명
2. 핵심 시각화 영역
3. 사용자 조작 영역
4. 실행 버튼
5. 현재 상태 또는 결과 해석 영역

권장 버튼 체계는 다음과 같다.

- 1회 실행
- 단계 실행
- 10회 실행
- 100회 실행
- 자동 실행
- 일시정지
- 초기화

개념에 따라 모든 버튼이 필요하지 않다면 생략해도 되지만, 반복 실험·누적·분포 형성 앱에서는 가능한 한 포함한다.

## 6-1. 학습지와 다운로드 자료 탑재

웹앱에 맞는 학습지를 함께 제작하거나 이미 함께 관리하는 경우, 사용자가 별도로 말하지 않아도 학습지 파일을 웹앱 폴더에 함께 두고 웹앱 화면에서 다운로드할 수 있게 연결한다.

- 웹앱과 직접 연결되는 학습지 원본 Markdown은 `tasks/`가 아니라 해당 웹앱 폴더에 둔다.
- PDF를 제작한 경우 PDF도 같은 웹앱 폴더에 두고 다운로드 링크를 만든다.
- 학생용과 교사용을 함께 제작한 경우 두 종류 모두 웹앱에서 받을 수 있게 한다.
- 다운로드 링크는 실제 존재하는 파일만 연결한다.
- 파일명은 영어 소문자, 숫자, 하이픈만 사용한다.
- 자료 다운로드 UI는 제목/짧은 설명 아래, 조작 영역 위에 얇게 배치한다.
- 자료가 1~2개이면 버튼을 바로 노출하고, 3개 이상이면 `<details>` 기반의 접이식 "자료 다운로드" 영역으로 묶는다.
- 자료 다운로드 영역이 핵심 시각화와 조작 영역을 밀어내지 않도록 작고 일관되게 유지한다.

권장 파일 예:

```text
probability-statistics/sample-proportion-event-readiness/
├─ index.html
├─ worksheet.md
├─ worksheet-teacher.md
├─ worksheet.pdf
└─ worksheet-teacher.pdf
```

권장 배치 예:

```html
<details class="resource-menu">
  <summary>자료 다운로드</summary>
  <div class="resource-list">
    <a href="./worksheet.pdf" download>학생용 PDF</a>
    <a href="./worksheet-teacher.pdf" download>교사용 PDF</a>
    <a href="./worksheet.md" download>학생용 MD</a>
    <a href="./worksheet-teacher.md" download>교사용 MD</a>
  </div>
</details>
```

## 7. 수학 웹앱 설계 원칙

웹앱은 단순 계산기가 아니라 수학적 변화 과정을 보여주는 도구여야 한다.

다음을 항상 고려한다.

- 사용자가 값을 바꾸면 무엇이 즉시 변하는가?
- 그래프, 도식, 표, 수치, 설명 문구가 함께 반응하는가?
- 학생이 현재 상태를 한눈에 읽을 수 있는가?
- 최종 결과만 보여주지 않고 중간 과정, 누적 변화, 수렴 또는 분포 형성을 보여주는가?
- 조작 결과가 수학 개념과 직접 연결되는가?
- 교사가 수업 중 짧게 시연하기에 충분히 단순한가?
- 그래프는 필요할 때만 사용하고, 사용자가 무엇을 봐야 하는지 분명해야 한다.
- 그래프는 최종 결과보다 변화 과정, 비교, 누적, 수렴, 분포 형성을 보여주는 데 사용한다.
- 하나의 화면에 너무 많은 그래프를 동시에 넣기보다 핵심 변화가 잘 보이게 구성한다.

피해야 할 것:

- 결과 숫자만 출력하는 앱
- 인터랙션이 거의 없는 앱
- 의미가 불분명한 그래프
- 한 화면에 너무 많은 요소를 넣는 구성
- 보기에는 화려하지만 수학적 의미가 약한 효과
- 학생이 같은 작업을 지나치게 반복해야만 이해되는 구조

## 8. 우선 다룰 수학 범위

### 확률과 통계

- 경우의 수
- 확률의 뜻
- 확률의 덧셈정리
- 조건부확률
- 사건의 독립과 종속
- 확률변수와 확률분포
- 기댓값과 표준편차
- 이항분포
- 정규분포
- 모집단과 표본
- 추정

### 미적분

- 수열의 극한
- 급수
- 삼각함수의 극한 (sinx/x 극한, 단위원 활용)
- 삼각함수의 도함수 (sin, cos, tan, sec, csc, cot)
- 지수함수와 로그함수 (자연현상과의 연결)
- 도함수의 활용
- 여러 가지 적분법
- 정적분의 활용

계산 결과보다 개념의 의미와 과정이 보이도록 구성한다.

## 8-1. 수학 표현 원칙

- 복잡한 수식은 LaTeX 문법으로 작성하고, MathJax 또는 MathML로 브라우저에서 읽기 좋게 렌더링한다.
- 단순한 인라인 수식은 텍스트 표기만으로 충분한지 먼저 판단한다.
- 학습지 Markdown 파일(`worksheet.md`, `worksheet-*.md` 등)에 들어가는 모든 수식은 반드시 LaTeX 형식으로 작성한다.
- 학습지 Markdown의 인라인 수식은 `$...$`, 별도 줄 수식은 `$$...$$` 형식을 사용한다.
- 학습지 Markdown을 제작할 때는 학생용 학습지와 예시답안이 있는 교사용 학습지를 함께 제작한다.
- 분수, 극한, 합, 적분, 확률식처럼 구조가 중요한 수식은 LaTeX 렌더링을 우선 고려한다.
- 분수는 `1/2`, `(X-np)/σ`, `a/b`처럼 슬래시 텍스트로만 두지 않고 교과서식 세로 분수로 렌더링되게 한다.
- 동적으로 바뀌는 수식은 조작 후 최신 값으로 갱신되고 다시 렌더링되게 한다.
- 화면에 원시 LaTeX 코드가 그대로 노출되지 않게 한다.
- MathJax를 사용할 때는 CDN 의존성이 생긴다는 점을 고려하고, 필요한 페이지에만 추가한다.
- 변수, 매개변수, 함수 이름, 확률기호, 극한기호는 한 웹앱 안에서 일관되게 사용한다.
- 수식만 나열하지 말고 그래프, 도식, 표, 현재 상태 문구와 연결한다.
- 수학적 설명은 정확하되 고등학생이 읽을 수 있는 문장으로 쓴다.

---

## 9. 코드 작성 원칙

- 전체 코드는 바로 실행 가능한 완성본으로 작성한다.
- HTML, CSS, JavaScript를 읽기 쉽게 작성한다.
- 변수명과 함수명은 의미가 드러나게 작성한다.
- 수학 계산 로직과 화면 갱신 로직을 가능하면 분리한다.
- 필요한 부분에는 짧은 주석을 단다.
- 자동 실행, 일시정지, 초기화 기능에서 상태가 꼬이지 않게 한다.
- 입력값 오류나 범위 초과를 고려한다.
- 외부 라이브러리는 꼭 필요한 경우에만 CDN으로 사용한다.
- CDN을 사용할 때는 사용 이유를 코드 주석이나 설명에 간단히 남긴다.
- 가능하면 Canvas, SVG, 기본 DOM 조작만으로 구현한다.

## 10. 접근성과 수업 사용성

다음을 지킨다.

- 버튼과 슬라이더는 충분히 크게 만든다.
- 숫자 결과는 단위나 의미를 함께 표시한다.
- 색만으로 의미를 구분하지 말고 텍스트나 모양도 함께 사용한다.
- 중요한 변화는 설명 문구로도 알려준다.
- 수업 시연용으로 처음 보는 학생도 흐름을 이해할 수 있게 한다.
- 화면 검토의 1순위 기준 기기는 iPad A16이다. Apple 공식 사양의 11형 2360×1640 디스플레이를 기준으로, 브라우저 검증에서는 가로 `1180×820` CSS px를 최우선으로 확인하고 세로 `820×1180` CSS px도 깨지지 않아야 한다.
- iPad A16 가로 화면에서 핵심 슬라이더, 버튼, 그래프 또는 도식, 현재 결과, 짧은 해석 문구가 첫 화면 또는 아주 짧은 스크롤 안에서 연결되게 한다.
- 교사가 프로젝터나 전자칠판으로 보여줄 때 조작 영역과 핵심 결과 시각화가 한 화면 안에 함께 보이게 한다.
- 핵심 슬라이더, 버튼, 그래프 또는 도식, 현재 결과, 짧은 해석 문구가 첫 화면 또는 아주 짧은 스크롤 안에서 연결되게 한다.
- 수업 시연 중 조작 영역과 결과 영역을 오가느라 과도한 스크롤이 필요한 배치는 피한다.
- 모바일보다 교실 PC와 프로젝터 환경을 우선하되, 모바일에서도 기본 조작은 가능하게 한다.

## 11. 목록 페이지 관리 규칙

새 웹앱을 추가할 때는 반드시 다음을 함께 수정한다.

1. 해당 웹앱 폴더 생성
2. 해당 폴더 안에 `index.html` 생성
3. 과목별 `index.html` 목록에 카드 추가
4. 루트 `index.html`의 전체 현황이 필요하면 함께 수정
5. 준비 중인 웹앱은 실제 파일이 없으면 링크하지 않는다

준비 중인 항목은 다음처럼 표시한다.

```html
<span class="status">준비 중</span>
```

실제 페이지가 없는 링크는 만들지 않는다.

## 13. GitHub Pages 배포 기준

GitHub Pages 배포를 전제로 하므로 다음을 지킨다.

- 모든 링크는 상대경로로 작성한다.
- 로컬에서 동작하더라도 GitHub Pages에서 깨질 수 있는 절대 로컬 경로를 사용하지 않는다.
- 다음과 같은 경로는 사용하지 않는다.

```html
<link rel="stylesheet" href="C:/Users/.../style.css">
```

- 다음처럼 상대경로를 사용한다.

```html
<link rel="stylesheet" href="../../assets/common.css">
```

- 각 폴더는 `/folder-name/` 형태로 접근해도 열리도록 `index.html`을 둔다.

## 15. Codex 작업 방식

Codex는 다음 방식으로 작업한다.

- 먼저 현재 폴더 구조를 확인한다.
- 기존 파일을 수정하기 전에 역할과 연결 구조를 파악한다.
- 새 웹앱을 만들 때 전체 구조를 깨지 않는다.
- 이미 있는 공통 스타일을 최대한 재사용한다.
- 하나의 요청에서 지나치게 많은 웹앱을 한꺼번에 만들지 않는다.
- 사용자가 요청한 웹앱 하나를 완성도 있게 만드는 것을 우선한다.
- 작업 후 수정한 파일 목록과 확인해야 할 점을 요약한다.

사용자가 명시하지 않는 한 하지 말아야 할 것:

- 프로젝트를 React, Vite, Next.js로 전환
- 패키지 설치
- 빌드 과정 추가
- 서버 기능 추가
- 기존 폴더 구조 대폭 변경
- 불필요한 외부 라이브러리 추가
- 실제 파일이 없는 링크 생성

## 16. 웹앱 완성 전 자체 점검

각 웹앱을 만든 뒤 다음 기준으로 스스로 점검한다.

- 이 앱이 보여주는 수학적 변화는 무엇인가?
- 사용자가 조작하면 어떤 요소가 즉시 바뀌는가?
- 그래프나 도식은 왜 필요한가?
- 학생이 현재 상태를 한눈에 읽을 수 있는가?
- 초기화와 반복 실행이 안정적인가?
- 자동 실행 중 중복 타이머가 생기지 않는가?
- 설명 없이도 사용 흐름을 어느 정도 이해할 수 있는가?
- 교사가 수업에서 바로 사용할 만큼 가독성이 충분한가?
- 모바일 화면에서 주요 버튼과 시각화가 깨지지 않는가?
- GitHub Pages 상대경로가 올바른가?

## 17. 완료 전 검증

작업을 마치기 전에 다음을 확인한다.

- 모든 링크가 실제 파일 또는 폴더를 가리키는가?
- 실제 파일이 없는 준비 중 항목은 링크가 아니라 `article.card`와 `.status`로 표시했는가?
- GitHub Pages에서 깨지는 로컬 절대경로가 없는가?
- 브라우저 콘솔 오류가 없는가?
- 버튼, 슬라이더, 초기화, 자동 실행이 동작하는가?
- 자동 실행 중 중복 타이머 또는 멈추지 않는 애니메이션이 생기지 않는가?
- iPad A16 가로 화면(`1180×820` CSS px)에서 핵심 조작과 결과 시각화가 첫 화면 또는 아주 짧은 스크롤 안에 함께 보이는가?
- iPad A16 세로 화면(`820×1180` CSS px)에서 주요 버튼, 수식, 그래프, 결과 카드가 겹치거나 잘리지 않는가?
- 분수와 구조가 중요한 수식이 교과서식으로 렌더링되고 원시 LaTeX가 보이지 않는가?
- 조작 후 동적 수식이 최신 값으로 갱신되는가?
- 프로젝터/전자칠판 화면에서 조작 영역과 핵심 결과 시각화가 한 화면 안에 함께 보이는가?
- 모바일과 교실 프로젝터 화면에서 주요 요소가 깨지지 않는가?
- 새 웹앱이 필요한 과목별 `index.html`에 등록되었는가?
- CDN을 사용했다면 사용 이유와 오프라인 제약을 고려했는가?

## 18. 웹앱 요청을 받았을 때의 기본 작업 순서

사용자가 새 웹앱 제작을 요청하면 다음 순서로 작업한다.

### 압축 설계 단계

1. 학습 목표를 1~2문장으로 정리한다.
2. 학습설계/작업지시서 에이전트 역할은 내부적으로 수행하고, 사용자에게는 `목표`, `조작`, `변화`, `오개념 방지`, `파일` 5항목 이내의 설계 요약만 보여준다.
3. 사용자가 기획 검토만 요청했거나 방향 확인 후 진행하라고 한 경우에는 여기서 멈춘다.
4. 사용자가 구현을 요청한 경우에는 압축 설계 요약 뒤 바로 실행 단계로 진행한다.

상세 작업지시서 전문은 사용자가 요청하거나 복잡한 수업 자료를 설계할 때만 작성한다.

### 실행 단계 (한 대화 안에서 완결)

구현 요청이 명확하면 이후 단계는 사용자의 추가 확인 없이 한 대화 안에서 연속으로 완료한다.

5. 학습가치 검토와 수학 표현 검토를 내부적으로 수행하고, `통과` 또는 `부분 수정 후 통과 가능`일 때만 구현한다.
6. 판정이 `재설계` 또는 `제작 보류`이면 사용자에게 짧은 판정과 이유를 알리고 작업지시서 단계로 돌아간다. 이 경우에만 실행 단계가 중단된다.
7. 프로젝터/전자칠판에서 조작과 결과가 한 화면 안에 함께 보이도록 화면 구성을 설계한다.
8. 필요한 파일을 생성하거나 수정한다.
9. 개별 웹앱 `index.html`을 완성한다.
10. 학습지를 함께 제작한 경우 웹앱 폴더에 Markdown/PDF를 함께 두고 다운로드 영역을 연결한다.
11. 과목별 목록 페이지에 링크를 추가한다.
12. iPad A16 가로/세로 화면, 수식 렌더링, 동적 갱신, 전자칠판 화면 배치, 버튼/슬라이더/초기화/자동 실행을 검증한다.
13. 사후 수학 표현 검토와 학습효과 검토는 짧은 판정으로 남긴다.
14. 로컬 확인 방법과 GitHub Pages 주소 예시를 알려준다.

## 19. 좋은 웹앱의 기준

이 프로젝트에서 좋은 웹앱은 다음과 같다.

- 정답을 보여주는 앱이 아니라 수학이 변화하는 모습을 보여주는 앱
- 학생이 직접 조작하며 패턴을 발견할 수 있는 앱
- 교사가 수업 중 짧게 보여줘도 핵심이 전달되는 앱
- 결과, 과정, 해석이 한 화면 안에서 연결되는 앱
- 코드가 단순하고 장기적으로 유지보수하기 쉬운 앱

최종 목표는 "웹에서 바로 실행되는 수학 개념 실험실"을 만드는 것이다.

## 20. 유지보수 기준

- 기본은 개별 앱이 자기 폴더의 `index.html` 하나만으로 실행되는 구조를 유지한다.
- 반복되는 버튼 상태 관리, 캔버스 헬퍼, 입력 검증처럼 여러 앱에서 재사용되는 작은 기능은 `assets/common.js`로 분리할 수 있다.
- 공통화하더라도 빌드 도구나 프레임워크를 도입하지 않는다.
- 큰 앱을 수정할 때는 수학 계산 로직, 상태 관리, 화면 렌더링 함수의 역할이 섞이지 않도록 한다.
- 이미 공개된 앱의 URL이 바뀌는 폴더명 변경은 피한다.
